# Chronos 复现实验操作指南

本文档不是 README，用于记录本项目的实际执行方式。当前推荐目标是：在 V100 单卡约 10 小时的限制下，尽量得到一组可汇报的 Chronos 轻量复现结果。

## 1. 复现目标

完整 Chronos 论文训练使用 10M TSMixup + 1M KernelSynth、200K steps、有效 batch size 256，并在多数据集 benchmark 上比较 WQL / MASE。V100 单卡 10 小时不足以完整复现 Small 或完整 200K 训练。

本项目采用更现实的目标，并把 Tiny 和 Mini 分成两条脚本：

```text
主实验 Tiny：Chronos-T5-Tiny + 官方训练语料 v100 子集 + 50K steps
辅助实验 Mini：Chronos-T5-Mini + 官方训练语料 v100 子集 + 10K steps，视剩余时间执行
对照实验：Seasonal Naive baseline + 官方 Chronos checkpoint zero-shot
```

这样可以验证 Chronos 的核心训练范式、数据混合方式、checkpoint/断点能力，以及训练后相对基线的 forecasting 表现。

## 2. 数据策略

主线数据来自 HuggingFace 官方 Chronos 训练语料：

```text
dataset_name: autogluon/chronos_datasets

TSMixup:
  training_corpus_tsmixup_10m

KernelSynth:
  training_corpus_kernel_synth_1m
```

混合比例保持论文设定：

```text
TSMixup : KernelSynth = 9 : 1
```

训练运行时会显式记录：

```text
dataset_name
data_level
training_data_paths
probability
```

记录位置：

```text
training_preflight.json
training_progress.json
checkpoint-final/training_info.json
```

## 3. 数据级别

| data_level | TSMixup | KernelSynth | Total | 用途 |
|---|---:|---:|---:|---|
| debug | 900 | 100 | 1,000 | 检查下载、转换、训练链路 |
| conservative | 45,000 | 5,000 | 50,000 | 网络或预算紧张时使用 |
| recommended | 90,000 | 10,000 | 100,000 | T4 / 本地较小实验 |
| stretch | 180,000 | 20,000 | 200,000 | T4 较完整实验 |
| v100 | 900,000 | 100,000 | 1,000,000 | V100 10 小时主实验 |

说明：`v100` 不是论文 full corpus，但保持官方数据源和 9:1 比例，规模是 `recommended` 的 10 倍，适合 V100 单卡时间限制。

## 4. 环境准备

```bash
python --version
pip install --upgrade pip
pip install -e ".[dev,test]"
```

如果需要从 `checkpoint-xxxxx` 完整续训，云端 torch 版本建议为 `>=2.6`。较新的 Transformers 会因为 `torch.load` 安全限制，拒绝在 torch `<2.6` 时加载 `optimizer.pt`。检查方式：

```bash
python -c "import torch; print(torch.__version__)"
```

如果版本低于 2.6，优先在云端环境中升级 torch，或者退而求其次使用“只从模型权重继续训练”：

```bash
python scripts/training/train.py \
  --config scripts/training/configs/chronos_t5_tiny_v100.yaml \
  --model-id outputs/checkpoints/chronos_t5_tiny_v100_50k/run-0/checkpoint-25000 \
  --no-random-init \
  --output-dir ./outputs/checkpoints/chronos_t5_tiny_v100_continue_from_weights
```

两者区别：

```text
resume_from_checkpoint:
  恢复模型、optimizer、scheduler、step，适合真正断点续训

model_id + no-random-init:
  只从模型权重继续，optimizer 和 scheduler 重新开始，适合作为保底方案
```

建议设置 HuggingFace 缓存，避免重复下载：

```bash
export HF_HOME=/home/ma-user/work/cache/huggingface
export HF_DATASETS_CACHE=/home/ma-user/work/cache/huggingface/datasets
export TRANSFORMERS_CACHE=/home/ma-user/work/cache/huggingface/transformers
export TOKENIZERS_PARALLELISM=false
```

确认 GPU：

```bash
nvidia-smi
python -c "import torch; print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0))"
```

## 5. 云端后台运行建议

长任务建议放在 `tmux` 或 `nohup` 中执行，避免终端断开导致进程退出。

`tmux` 示例：

```bash
tmux new -s chronos
bash scripts/cloud/debug_cloud.sh

# 断开但保留任务
Ctrl-b d

# 重新进入
tmux attach -t chronos
```

`nohup` 示例：

```bash
nohup bash scripts/cloud/tiny_v100_train.sh > outputs/cloud_logs/tiny_v100_nohup.log 2>&1 &
tail -f outputs/cloud_logs/tiny_v100_nohup.log
```

训练日志和诊断文件会分别保存在：

```text
outputs/cloud_logs/*.log
outputs/checkpoints/*/run-*/training_preflight.json
outputs/checkpoints/*/run-*/training_progress.json
outputs/checkpoints/*/run-*/training_failure.json
```

```bash
python scripts/data/inspect_arrow_dataset.py \
```

```bash
bash scripts/cloud/debug_cloud.sh
```

## 7. V100 数据准备

V100 主实验数据：

```bash
python scripts/data/prepare_chronos_corpus_subset.py \
  --output_dir data/processed \
  --data_level v100 \
  --min_length 576 \
  --max_length 2048 \
  --seed 42 \
  --compression lz4
```

输出文件：

```text
data/processed/chronos_tsmixup_900k.arrow
data/processed/chronos_kernel_synth_100k.arrow
data/processed/chronos_corpus_subset_summary.json
data/processed/chronos_corpus_subset_progress.json
```

检查数据：

```bash
python scripts/data/inspect_arrow_dataset.py \
  --paths data/processed/chronos_tsmixup_900k.arrow data/processed/chronos_kernel_synth_100k.arrow \
  --freq h \
  --min_length 576 \
  --max_examples 5
```

`chronos_corpus_subset_summary.json` 必须保留，用于报告数据来源、数量、过滤统计和 sha256。

数据准备脚本是流式写入 Arrow 文件，不会把 100 万条序列一次性攒入内存。如果中途失败，临时文件会删除；已完成的 TSMixup 或 KernelSynth 文件会在下次运行时跳过，除非显式传入 `--overwrite`。

## 8. V100 Tiny 主实验

配置文件：

```text
scripts/training/configs/chronos_t5_tiny_v100.yaml
```

核心参数：

```text
model_id: google/t5-efficient-tiny
data_level: v100
context_length: 512
prediction_length: 64
max_steps: 50000
per_device_train_batch_size: 16
gradient_accumulation_steps: 8
effective_batch_size: 128
fp16: true
tf32: false
torch_compile: false
```

运行：

```bash
bash scripts/cloud/tiny_v100_train.sh
```

如果前 1000 steps 很快，并且剩余时间充足，可以覆盖步数继续跑：

```bash
python scripts/training/train.py \
  --config scripts/training/configs/chronos_t5_tiny_v100.yaml \
  --max-steps 100000 \
  --save-steps 10000 \
  --output-dir ./outputs/checkpoints/chronos_t5_tiny_v100_100k
```

如果云端中断，找到最近 checkpoint 后续跑：

```bash
export RESUME_FROM_CHECKPOINT=outputs/checkpoints/chronos_t5_tiny_v100_50k/run-0/checkpoint-25000
bash scripts/cloud/tiny_v100_train.sh
```

注意：续跑会继续写入当前配置指定的输出目录；报告中必须记录续跑来源 checkpoint。

## 9. V100 Tiny 评估

Tiny 是主实验，评估也优先保证完整。默认策略：

```text
checkpoint-10000 / checkpoint-25000 / checkpoint-50000 / checkpoint-final:
  quick-zero-shot
  quick-in-domain

checkpoint-final:
  zero-shot
  in-domain
```

运行：

```bash
bash scripts/cloud/tiny_v100_eval.sh
```

如果要指定某个 run：

```bash
bash scripts/cloud/tiny_v100_eval.sh outputs/checkpoints/chronos_t5_tiny_v100_50k/run-0
```

如果时间不够，只跑快速评估：

```bash
export RUN_FULL_FINAL_EVAL=0
bash scripts/cloud/tiny_v100_eval.sh
```

评估输出：

```text

outputs/evaluation/tiny_v100_final/
```

每个目录中会保存：


## 11. 本地评估与后处理

模型训练后的评估可以在本地运行，只要把云端 checkpoint 目录和 `outputs/evaluation/` 下载到本地。但正式评估更推荐在云端完成，因为评估也是模型推理，本地 3060 会慢很多。

推荐分工：

```text
云端：
  训练 Tiny / Mini
  对关键 checkpoint 做 quick eval
  对 Tiny final 做 full eval

本地：
  检查 CSV 完整性
  汇总 checkpoint 指标变化
  画 loss / MASE / WQL 图
  写报告
  必要时对单个 checkpoint 做 sanity eval
```

本地评估示例：

```bash
python scripts/evaluation/run_checkpoint_evaluation.py \
  --run_dir outputs/checkpoints/chronos_t5_tiny_v100_50k/run-0 \
  --model_name tiny_v100_local_check \
  --checkpoints checkpoint-final \
  --suites quick-zero-shot \
  --device cuda:0 \
  --torch_dtype float16 \
  --batch_size 4
```

如果本地显存不够，改用：

```bash
--device cpu --torch_dtype float32 --batch_size 1
```

评估完成后，可以和 Seasonal Naive 基线比较：

```bash
python scripts/evaluation/compare_to_baseline.py \
  --model_csv outputs/evaluation/tiny_v100_final/tiny_v100_final-checkpoint-final-zero-shot.csv \
  --baseline_csv scripts/evaluation/results/seasonal-naive-zero-shot.csv \
  --output_csv outputs/evaluation/tiny_v100_final/tiny_vs_seasonal_zero_shot.csv \
  --output_json outputs/evaluation/tiny_v100_final/tiny_vs_seasonal_zero_shot.json
```

`relative_MASE` 或 `relative_WQL` 小于 1，表示该模型在对应指标上优于 Seasonal Naive。

## 12. 日志与故障排查

训练脚本会生成：

```text
training_preflight.json
training_progress.json
training_failure.json
```

用途：

```text
training_preflight.json  记录配置、环境、GPU、数据路径和文件大小
training_progress.json   记录当前训练阶段
training_failure.json    失败时记录异常类型、异常信息和 traceback
```

云端失败时优先查看：

```bash
cat outputs/checkpoints/*/run-*/training_progress.json
cat outputs/checkpoints/*/run-*/training_failure.json
cat outputs/checkpoints/*/run-*/training_preflight.json
```

常见处理：

```text
OOM:
  per_device_train_batch_size: 16 -> 8 -> 4
  gradient_accumulation_steps: 8 -> 16 -> 32
  gradient_checkpointing: false -> true

速度太慢:
  先保留 Tiny 50K
  Mini 只跑 5K-10K

数据下载慢:
  保留 chronos_corpus_subset_progress.json
  确认 HF_HOME / HF_DATASETS_CACHE 在持久化目录

评估失败:
  查看 outputs/evaluation/*/evaluation_failure.json
  降低 EVAL_BATCH_SIZE
  本地评估时使用 --device cpu --torch_dtype float32
```

## 13. 推荐执行顺序

```bash
# 1. 安装环境
pip install -e ".[dev,test]"

# 2. 准备 debug 数据
python scripts/data/prepare_chronos_corpus_subset.py --data_level debug --output_dir data/processed --overwrite

# 3. 检查 debug 数据
python scripts/data/inspect_arrow_dataset.py \
  --paths data/processed/chronos_debug_tsmixup_900.arrow data/processed/chronos_debug_kernel_synth_100.arrow

# 4. Debug 训练
python scripts/training/train.py --config scripts/training/configs/chronos_t5_tiny_debug.yaml

# 5. Tiny 主实验，包括 V100 数据准备、检查、训练
bash scripts/cloud/tiny_v100_train.sh

# 6. Tiny 评估
bash scripts/cloud/tiny_v100_eval.sh

# 7. 如果还有时间，再执行 Mini
bash scripts/cloud/mini_v100_train_optional.sh
bash scripts/cloud/mini_v100_eval_optional.sh
```

不要提交 `data/` 和 `outputs/` 到 Git；它们已在 `.gitignore` 中忽略。

## 14. 报告中应如何表述

推荐表述：

```text
在 V100 单卡约 10 小时预算下，本实验使用 Chronos 官方训练语料的 1M 子集，保持 TSMixup:KernelSynth = 9:1、context length 512、prediction length 64，训练 Chronos-T5-Tiny 50K steps，并以 Mini 10K steps 作为模型规模探针。该实验不等同于论文 11M corpus + 200K steps 的完整复现，但可以验证 Chronos 训练范式、数据混合策略和轻量模型在 zero-shot forecasting 上相对基线的趋势。
```
